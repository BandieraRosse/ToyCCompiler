# ToyCCompiler (tcc)

> *"Yo sé quién soy."*  
> — 米格尔·德·塞万提斯，《堂吉诃德》

一个面向 x86-64 Linux 的自举 C 编译器。约一万行 C 代码，零外部依赖，零 libc——只有一条 `syscall` 指令和编译自身的意志。

## 这是什么

ToyCCompiler 是一个小型的、独立的 C 编译器，能够编译自己的源代码。它不是任何项目的分支。它不使用 LLVM、GCC 或任何已有的编译器基础设施。它从几个 C 源文件出发，最终产出一个可运行的 ELF 二进制——全程没有一条 `#include <stdlib.h>`。

运行时（`tcc_rt.c`）通过内联 `syscall` 指令直接与 Linux 内核通信。输出为静态链接的 ELF64 `.o` 文件。测试套件运行 28 个自包含的测试程序，这些程序不需要任何运行时库。

## 自举链

```
gcc  ──→  stage-1 tcc
               │
               ↓  用自身编译 tcc 源码
          stage-2 tcc
               │
               ↓  用自身编译 tcc 源码
          stage-3 tcc
               │
               ↓  ...
          stage-4 tcc   ──→  与 3、5、6、7、8、9、10 字节完全一致
```

| 阶段 | 编译者        | 大小     | Selfhost 测试 | MD5                              |
|------|---------------|----------|---------------|----------------------------------|
| 1    | gcc（宿主）    | 202,400  | 28/28 ✅      | （由 GCC 签名）                   |
| 2    | stage-1 tcc   | 349,544  | 28/28 ✅      | `f80002c7`                       |
| 3–10 | stage-2 tcc+  | 349,544  | 28/28 ✅      | `68de925f`（全部相同）            |

结果：**Stage-3 到 Stage-10 字节级完全一致。** 编译器已经到达其不动点——一个自我复制的、封闭的代码回路，在每一次迭代中都忠实地生成自身。

Stage-2 与 Stage-3 之间唯一 8 字节的差异（位于 `lex.o` 的两个函数中）是一次交接的签名：Stage-2 由 GCC 编译，Stage-3 由 tcc 自身编译。此后链条收敛，不再变化。

## 代码风貌

```c
static long sys_write(int fd, const char *buf, unsigned long len) {
    unsigned long ret;
    __asm__ __volatile__ ("syscall"
        : "=a"(ret) : "a"(1), "D"((long)fd), "S"((long)buf), "d"((long)len)
        : "rcx", "r11", "memory");
    return (long)ret;
}

void __tlibc_start(void) {
    sys_write(1, "Hello, world.\n", 14);
    __asm__ ("syscall" : : "a"((long)60), "D"((long)0) : "rcx", "r11");
}
```

没有 `main()`。没有 `return`。没有 libc。入口点是 `__tlibc_start`，测试程序通过原始的 `exit` 系统调用退出。内联汇编约束有意保持简单——只有 `"=a"` 和 `"a"`——因为编译器的汇编支持在设计上是最简的。

## 为什么做这个

C 编译器并不稀缺。GCC 有一千五百万行代码。LLVM 是一个庞大的生态系统。即使是 Fabrice Bellard 的 `tcc` 也比这个庞大十倍。

做这个的原因有两个：

1. **为了看看能不能做到**——用最少的代码行，零依赖、零运行时、零外部帮助，写一个自举的 C 编译器。只有源码、CPU 和 System V ABI。

2. **为了彻底地理解**——你自己写的编译器，你自己就能完全理解。每一个 Bug 都是你的。每一个取舍都是你的。每一行最终正确的 `mov %rdi,-0x8(%rbp)` 都是你的。

## 名字

*ToyCCompiler*（玩具 C 编译器）诚实地承认了自己的局限。这个编译器不奢望取代 GCC，也不指望随 Linux 发行版一起发布。它期望的是编译自身，证明自己的一致性，并作为一个完整的、可读的作品，展示 C 语言是如何变成机器码的。

这个项目的守护者是堂吉诃德——不是因为他与风车搏斗，而是因为他选择了一条清醒的、孤独的、属于自己的路。

## 旅程中的一段注记

那个阻断了完整自举的 Bug 小得令人意外：`parse_parameter_list` 中一个 `Lexer` 结构体复制只复制了 8 字节而不是 88 字节。结果是 `void *dst` 参数被悄然截断为 32 位——生成的指令是 `%edi` 而非 `%rdi`。Stage-1（由 GCC 编译）没有问题。Stage-2（由 Stage-1 编译）无声地继承了 Bug。Stage-3（由 Stage-2 编译）崩溃了。

这就是经典的*可信信任*问题变得可见的时刻：编译器的 Bug 可以隐藏在自己编译出的二进制中，直到链条在第三代时才会暴露。

修复是一处函数改动。验证是 Stage-1 到 Stage-10。

## 局限

- 仅支持 x86-64 Linux
- C89 加上部分 C99 特性（不支持 VLA、位域、`_Generic`）
- 默认构建不支持浮点数
- `main()` 的 return 语句不可靠——请使用 `sys_exit()` 代替
- 变参调用限制在约 6 个参数以内
- `*(p + i)` 形式的指针运算不可靠——请使用 `p[i]`

详细的构建和测试说明请参见 [`CLAUDE.md`](./CLAUDE.md)。

## 验证

```sh
make test           # 28/28  ✅
make test-selfhost  # 28/28  ✅
./bootstrap-to-10.sh  # stage-3 → stage-10 字节完全一致 ✅
```

全部 28 个自包含测试在 Stage-1 到 Stage-10 的每一个阶段均通过。编译器的输出在 Stage-3 收敛，并在 Stage-3 到 Stage-10 之间保持完全一致——一次完整而可验证的自举。
